# r1 — cycle-028 delivery review (B-030 macOS sandbox-exec containment)

Verdict: findings. I reviewed the committed `crates/symphony-grade` delta and the cycle-028 outcome/Grade/fix artifacts; I ignored the unrelated uncommitted `scripts/verify-docs.sh` change.

The SBPL rule itself is not the main escape: when the trusted macOS `sandbox-exec` is actually used, `sandbox.rs` builds a default-allow profile with `deny file-read*` over canonical forbidden roots, and `paths.rs` routes command acceptances through that wrapper plus `setsid`. The escaped issues are around proof and invocation trust: the core containment tests can pass without an active sandbox, and production checks for `/usr/bin/sandbox-exec` but then invokes `sandbox-exec` through `PATH`.

## Findings

### F1 — P0 — macOS containment tests pass when sandbox is inactive

`outcome.md` requires the P0 proof test to run an obfuscated forbidden read “under the sandbox” and prove the secret never reaches stdout/redirect target (`outcome.md:115-119`, `185-194`). The merged test does that only if a preliminary probe is active. If the probe is inactive, it asserts only that an unavailable reason exists and returns success:

- `session.rs:1086-1090`: `let probe = ...; if !probe.active { assert!(probe.unavailable_reason.is_some()); return; }`
- same fail-open pattern for symlink containment at `session.rs:1134-1138`
- inactive probes are normal fallback outputs from `sandbox.rs:52-60`

The temp-dir fix is not itself the problem: on the active path, `session.rs:1091-1123` still checks external leak content, stdout, and `"sandbox_active": true`. The problem is that the test is not fail-closed. A broken/missing `sandbox-exec` or invalid profile can make the P0 containment test pass without running the hostile command under OS containment.

### F2 — P1 — trusted sandbox binary is discovered but not invoked by absolute path

`sandbox_exec_path()` checks for `/usr/bin/sandbox-exec` or `/bin/sandbox-exec` (`sandbox.rs:109-115`), but the result is discarded. Both the apply probe and production command use `Command::new("sandbox-exec")` / `program: "sandbox-exec"`:

- probe: `sandbox.rs:78-84`
- production wrapper: `sandbox.rs:62-69`

A fake `sandbox-exec` earlier in `PATH` can return success for the probe and then run `sh -c <command>` unsandboxed while the trace reports `sandbox_active: true`. That is a real containment false-sense under a manipulated Grade environment. Use the absolute path returned by `sandbox_exec_path()` for both the probe and production wrapper; preferably use an absolute `/bin/sh` too.
