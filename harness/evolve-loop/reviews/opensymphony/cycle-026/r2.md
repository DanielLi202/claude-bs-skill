## F1 gate analysis

a6 explicitly required: if `GET /api/v1/capabilities` or `GET /api/v1/config` rejects, the panel must name the failed GET and provide retry. The implementation’s escaped branch was: refresh POST succeeds, follow-up capabilities GET fails, UI mislabels it as the POST failure and provides no GET retry.

Grade missed it because:
- `grade_round_1.md` marked a6 pass with generic prose: “GET rejection renders honest error line…” but did not prove each named failure branch.
- `negative_regression_tests` in round 1 omitted a6 entirely.
- `grade_lint.py` v1.4.24 enforces block shape and some hardcoded property facets, but does not derive “named failure branch” obligations from acceptance prose.
- Its missing-negative check uses blocking IDs from outcome acceptance severity. The a6 outcome row has no severity field, while Grade’s matrix row supplies `severity_if_fail: P1`; therefore missing `negative_regression_tests[a6]` was not blocking.
- Existing frontend validators cover production wiring, pending dismissal, SSE reconnect, identity mismatch, terminal enum coverage, DA-30 shell retry/snapshot retention, UI-source affordances, schema/tag render, and pessimistic-command / 409-conflict-refetch cases. None covers generic HTTP GET failure branch coverage after a successful command/refetch.
- Shape contributed by packing initial load, POST re-probe, successful refetch, two GET rejection paths, and retry semantics into one a6 row with no machine-readable branch matrix.
