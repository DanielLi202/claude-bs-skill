## F1 — P1 — Re-probe mislabels failed capability refetches as POST failures and can leave the panel “loaded” without config

Acceptance a6 requires GET failures to name the failed GET and expose retry: outcome.md:142-143 says that when `GET /api/v1/capabilities` or `GET /api/v1/config` rejects, the panel renders an honest error line naming the failed call and a retry affordance.

The merged store collapses two operations into one catch:

- `apps/symphony-ui/src/lib/store/tweaksStore.ts:73-82`:
  - calls `client.refreshCapability()`
  - then calls `client.capabilities()`
  - any failure from either path lands in `reprobeError`

The UI then always renders that `reprobeError` as a POST failure:

- `apps/symphony-ui/src/tweaks/TweaksPanel.tsx:59`:
  - `POST /api/v1/daemon/commands/refresh-capability failed: {state.reprobeError}`

So if the POST succeeds but the required follow-up `GET /api/v1/capabilities` fails, the panel falsely says the POST failed and provides no retry affordance for the failed GET. This is not just copy: if initial config load had failed, clicking the always-visible re-probe button can set `load: "loaded"` with `config: null` (`tweaksStore.ts:78` preserves the null config), producing a loaded Tweaks panel without config source layers.

The test gap is visible in `apps/symphony-ui/src/tweaks/tweaks.test.tsx:157-181`: it tests a successful re-probe refetch and an initial mount GET failure, but never tests a failed capabilities refetch after a successful refresh POST.

Minimal fix: split refresh POST errors from capability-refetch GET errors; on refetch failure, render `GET /api/v1/capabilities failed: ...` with retry, and do not transition to `load: "loaded"` unless the config requirement remains satisfied.
