# r1 — cycle-027 delivery review (B-027 UI-M5 Conduct monitor)

Grade round 0’s F1/F2 were real, and round 1 fixed those narrow failures. The frontend gates were not skipped: the documented offline Conduct block is honest, and the orchestrator evidence shows pnpm test failed in round 0 and passed in round 1. The `transport.ts` one-line change is contract-legitimate NDJSON support, not a frozen-substrate violation.

But the merged deliverable still misses the production invalidation path it claims to implement. The tests prove mostly component rendering and mocked store/client behavior; they do not prove the app is wired to SSE progress/reconnect invalidations. NDJSON parsing is also still not robust for malformed/non-JSON lines.

## Escaped findings

### F1 (P1) — Production SSE invalidation/reconnect lifecycle is not wired, so a3/a4 are not truly met

The acceptance contract requires the monitor to fetch daemon progress **after** `state_changed{what:"progress"}` and to reflect the existing SSE reconnect/degraded lifecycle. The merged app never starts that lifecycle in production: `main.tsx` only renders `<App />`, `AppShell` only does an initial `fetchState(client)`, and a repo-wide non-test search for `subscribeToInvalidations` finds only the helper definition, not any production caller.

`ConductMonitor` fetches progress only on mount or when `run.run_id` / `run.revision` changes:

- `apps/symphony-ui/src/conduct/ConductMonitor.tsx:29-42` calls `client.runEvents(... { type: "daemon_progress" })`, but the effect is keyed only by `[client, run.run_id, run.revision]`.
- `apps/symphony-ui/src/lib/sse/connection.ts:161-167` handles `state_changed` by `refresh()`ing state, not by exposing the invalidation to this monitor.
- `apps/symphony-ui/src/main.tsx:6-8` renders `<App />` with no SSE controller setup.

This contradicts the normative API contract: `docs/architecture/api-contract.md:613-618` says slow ops emit `daemon_progress`, push `state_changed{what:"progress"}`, and the UI then calls `GET /runs/{id}/events?type=daemon_progress&since_seq=<n>`.

The tests mask the gap. `conduct.test.tsx:98-103` checks `latestProgressText()` with a hand-built fake SSE-shaped object, then renders `<App client={conductClient(... progressEvents(...))}>`; no `state_changed` invalidation is delivered. `conduct.test.tsx:115-124` manually calls `connectionStore.reconnecting()` / `.degraded()` instead of proving `subscribeToInvalidations` is mounted in the app. That is not a production-path anchor for DA31/DA30.

### F2 (P1) — `runEvents` still throws on malformed/non-JSON NDJSON lines

Round 1 fixed the single-line valid JSON object case, but the parser is still not complete for the NDJSON robustness neighborhood requested in review. `parseNdjsonEvents` parses each nonblank line with raw `JSON.parse`:

- `apps/symphony-ui/src/lib/api/client.ts:286-292`:
  - non-string inputs return `[]`;
  - blank lines are filtered;
  - but any malformed/non-JSON line throws out of `JSON.parse(line)`.

That means a body such as a valid `daemon_progress` line followed by `not-json`, or any transient corrupt line in the NDJSON endpoint, rejects `client.runEvents()` instead of safely ignoring/reporting the bad line while preserving valid events. The prompt explicitly required malformed/non-JSON lines to be handled without throwing, and Grade round 1 only covered single-line valid NDJSON/object coercion.
