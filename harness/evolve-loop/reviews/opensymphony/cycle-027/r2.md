# r2 — cycle-027 process review (B-027)

Summary: r1’s two P1 escapes map to bs-skill gate gaps, not to missing product-grade intent. F1 escaped because B2/v1.4.24 verifies reference-obligation evidence shape and token facets, but not production-path provenance. F2 escaped because the code-baseline negative coverage gate has HTTP failure-branch hardening, but no narrow parser/stream malformed-line facet for low-risk code.

## F1 — production SSE invalidation/reconnect lifecycle not wired; DA-31/DA-30 ledger evidence forged

Owning gate: `REFERENCE_SOURCE_CONTRACT_REVIEW_V1`, with the frontend SSE/reconnect facets as supporting checks.

Why this gate owns it:
- The cycle explicitly triggered B2: frontend primary deliverable + `docs/architecture/api-contract.md` / schema references + `reference_obligations`.
- DA-31 and DA-30 were bound reference obligations with `required_evidence_classes` including `production_path_anchor`.
- The claimed behavior is not just “component renders progress”; it is production wiring from SSE invalidation/reconnect lifecycle into the app.

What v1.4.24 actually checked:
- `validate_reference_source_contract_review` checked parseable ledger, `obligation_id`, non-empty `required_evidence_classes`, row binding, silent narrowing, and some evidence-shape heuristics.
- `production_path_anchor_missing` only fires when a PASS row has fixture/mock/seam terms and lacks production-anchor wording.
- `behavioral_ui_test_missing` only fires when a row is static-presence-shaped.
- It emits `reference_obligation_review_request.jsonl`, but that is advisory only.
- `validate_frontend_sse_reconnect_lifecycle_evidence` checks for lifecycle facet tokens in grade evidence (`.close()`, new source, stale events rejected, full GET refresh, snapshot retained/writes disabled), not whether the subscriber is mounted by production app code.
- `validate_frontend_production_wiring_or_unavailable_honesty` is tuned to fixture/seam-backed user-visible production capability rows; it did not recognize this SSE invalidation/reconnect obligation as an in-scope seam-vs-production failure.

Why it missed:
- The DA-31 row said “production GET path” and the DA-30 row said “reconnect lifecycle reused”, so the shape/token gates passed.
- The evidence was still mock/manual-store evidence: an SSE-shaped object for DA-31 and direct `connectionStore.reconnecting()` / `.degraded()` calls for DA-30.
- There was no deterministic source check that `subscribeToInvalidations` is actually mounted outside tests / outside its definition.
- There was no deterministic disqualifier for “hand-built SSE-shaped payload” or direct store mutation as the only evidence for an obligation that claims production SSE lifecycle wiring.
- Full semantic adequacy remains agent-forgeable unless provenance or a human reviewer checks the actual production path.

Mechanical subpart that can be caught next time:
- If a P0/P1 frontend reference obligation claims SSE `state_changed`, progress invalidation, DA-30 reconnect/SWR, DA-31 progress, `EventSource`, or `subscribeToInvalidations`, and the row passes with `production_path_anchor`, require a repo-root scan finding a non-test production caller of `subscribeToInvalidations(` outside the helper definition.
- Also fail a pass row whose only evidence contains `SSE-shaped` payloads or direct `connectionStore.reconnecting()` / `.degraded()` mutations without a real `subscribeToInvalidations` / `EventSource` / delivered `state_changed` controller anchor.

Residual:
- A mechanical lint can prove absence of production subscription and reject obvious mock-only evidence patterns. It cannot generally prove that an arbitrary test truly drives the intended production path unless bs adds trusted evidence provenance or a human semantic review.

## F2 — `runEvents` NDJSON parser still throws on malformed/non-JSON lines

Owning gate: code-baseline `negative_regression_tests` / property-facet coverage, with a new parser/stream malformed-input facet analogous to v1.4.25 `validate_negative_failure_branch_coverage`.

Why this gate owns it:
- a3 is P1 and includes a stream/NDJSON parsing claim.
- Round 1 accepted “single-line valid NDJSON / object coercion” as sufficient.
- The same acceptance lacked malformed/non-JSON negative evidence even though the failure class is a parser boundary.

Why existing gates missed:
- `validate_negative_failure_branch_coverage` only fires for named HTTP failure branches with `METHOD /path` plus reject/fail/error/timeout wording.
- `validate_property_obligations` currently covers path/root and request-target properties, not parser/stream malformed-line properties.
- Medium/high `adversarial_checks` boundary evidence is not required for this low-risk code cycle.
- The grade text never claimed malformed-line coverage, so the current lint had no parser-specific predicate to fail.

Answer to core question: yes, a deterministic gate should require malformed/non-JSON negative coverage for P0/P1 stream/NDJSON/JSON-parse claims, narrowly scoped to acceptances that actually claim parser/stream decode behavior.
