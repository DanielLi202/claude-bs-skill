### F7 — P1 — capsule model is not a schema-v1.2 guard beyond F4

`capsuleModel.ts` accepts many schema-invalid capsules as valid, not just the documented `llm_judge.criteria` array/string bug.

Evidence:
- Spec requires top-level fields including `schema_version`, `iteration`, `output_contract`, controlled `tags`, and valid IDs: `docs/architecture/schemas/outcome-capsule.md:24-43`.
- Spec says unknown tags reject: `docs/architecture/schemas/outcome-capsule.md:53-65`.
- Spec high-risk `action` is a closed enum: `docs/architecture/schemas/outcome-capsule.md:185-193`.
- Spec groundings require source-type-specific locator and `supports[] ≥ 1`: `docs/architecture/schemas/outcome-capsule.md:345-356`.
- Code silently treats missing required arrays as empty: `apps/symphony-ui/src/outcome/capsuleModel.ts:144-155` — `if (value === undefined) return [];`.
- Code parses tags only as strings, no controlled vocab: `capsuleModel.ts:121`, `269-275`.
- Code accepts any high-risk `action` string: `capsuleModel.ts:238-248`.
- Code accepts empty `supports` and any `url ?? path` regardless of `source_type`: `capsuleModel.ts:217-222`.
- Its “complete” fixture is itself schema-invalid: `apps/symphony-ui/src/outcome/outcome.test.tsx:390-418` has `id: "T-1"`, no `schema_version`, no `iteration`, no `output_contract`, and high-risk actions `"execute"` / `"second check"`.

Failure mode: the UI can present a malformed outcome capsule as trustworthy and executable.

Failure mode: a `tdd`, `exploratory`, `migration`, etc. capsule loses visible review context that affects downstream behavior.

### F9 — P1 — stale `RunDetail` can override fresher daemon state and revision

Evidence:
- Inspector lets detail override snapshot fields: `apps/symphony-ui/src/shell/Inspector.tsx:52` — `const displayRun = detail ? { ...run, ...detail } : run;`.
- The detail fetch effect is keyed only by run id/stage, not revision, and does not clear stale detail when the snapshot leaves `shape`: `apps/symphony-ui/src/shell/AppShell.tsx:45-56`.
- Execute uses detail revision first: `apps/symphony-ui/src/outcome/OutcomeActions.tsx:102-104` — `return detail?.revision ?? run.revision;`.
- Capsule explicitly forbids fabricated/stale stage truth: `.prompts/dogfood/cycle-025/outcome.md:17` — “UI MUST NOT fabricate stage transitions”.

Failure mode: after a state refetch reports `conduct.probing` or a newer revision, an older `RunDetail` can keep rendering `shape.outcome_ready` and send stale `If-Match`.

### F10 — P1 — `re-shape` continue/reject path is not implemented as PESSIMISTIC

Evidence:
- API classifies `re-shape mode=continue` as PESSIMISTIC with required `If-Match`: `docs/architecture/api-contract.md:637`, `648-651`.
- UI responsiveness requires scoped disable and inline failure: `docs/architecture/ui-responsiveness.md:184-190`.
- Design explicitly excludes modal but still requires scoped pessimistic: `docs/ux/design-brief.md:979-995`.
- Outcome action button calls `void reShape(...)` and shares only Execute’s pending state: `apps/symphony-ui/src/outcome/OutcomeActions.tsx:31-36`.
- `reShape` has no pending state, no inline error state, no catch, no scoped disable: `OutcomeActions.tsx:77-100`.
- Assumption reject also discards the promise: `apps/symphony-ui/src/outcome/OutcomeReadyCard.tsx:139-153`.

Failure mode: failed or slow re-shape requests produce unhandled rejections, repeated clicks, and no affected-region error.

### F11 — P1 — 409 conflict refetch failure strands Execute in pending state

Evidence:
- Execute marks pending before POST: `apps/symphony-ui/src/outcome/OutcomeActions.tsx:54`.
- In the conflict branch, it awaits `refreshAfterCommand` before setting conflict state: `OutcomeActions.tsx:67-71`.
- If that refetch fails, no `executeConflict` or `executeError` runs; caller discards the promise: `OutcomeActions.tsx:32`.
- PESSIMISTIC failure must be inline in the affected region: `docs/architecture/api-contract.md:648-651`.

Failure mode: a normal 409 followed by transient GET failure leaves the Execute region indefinitely pending.
